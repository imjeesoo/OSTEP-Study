# Context Switch Overhead Measurement on Linux  
  
## 실험 개요
운영체제가 실행 중인 프로세스를 중단하고 다른 프로세스로 CPU 제어권을 넘기는 Context Switch Overhead를 측정한다.  
두 개의 프로세스가 파이프(Pipe)를 통해 1바이트의 데이터를 주고받는 과정을 반복하게 하여, 강제적인 Context Switch를 유도하고 그 평균 비용을 산출한다.

## 실험 목적  
 - Context Switch의 평균 Overhead 측정: 프로세스 교체 시 발생하는 시간적 손실을 정량적으로 파악.  
 - 프로세스 상태 변화 이해: read/write 시 발생하는 Blocking과 실행 상태 전환 과정 학습.  
 - 정밀 시간 측정 방식의 이해: 나노초 단위 측정을 통해 아주 짧은 찰나의 커널 오버헤드를 계산하는 방법 숙지.   
  
## 실험 환경  
 - OS: Ubuntu  
 - 실행 환경: VMware Virtual Machine  
 - 언어: C  
 - 컴파일러: GCC  
  
## 측정 방법  
실험 과정은 다음과 같다.  
pipe()를 이용해 부모와 자식 간의 양방향 통신 채널(p1, p2)을 구축한다.  
fork()를 호출하여 부모와 자식 프로세스를 생성한다.  
부모가 데이터를 전송하고 자식의 응답을 기다리는 동안, 부모는 대기 상태가 되며 CPU는 자식에게 할당된다. (Context Switch 발생)  
이 과정을 입력받은 count만큼 반복한 뒤 전체 시간을 측정한다.  
계산 공식: 전체 시간 / (2 * count)  
데이터가 왕복할 때마다 부모→자식, 자식→부모로 총 2회의 문맥 전환이 발생하기 때문이다.  

**시간 측정 방식**  
clock_gettime(): POSIX 권장 방식으로, 나노초(ns) 단위의 정밀한 시간을 반환한다.  
문맥 전환은 마이크로초 단위 이하에서 발생할 수 있으므로, gettimeofday()보다 더 높은 정밀도를 가진 이 방식이 성능 측정에 필수적이다.  

## 실험 해석 및 정리  
이 실험에서 측정되는 값은 단순히 데이터를 옮기는 시간이 아니라, 다음과 같은 커널의 작업 비용을 포함한다.  
Register Save/Restore: 현재 실행 중인 프로세스의 레지스터 상태를 PCB(Process Control Block)에 저장하고  
다음 프로세스의 상태를 복구하는 과정.  
Scheduler 개입: 다음으로 실행할 프로세스를 결정하는 로직 수행.  
Mode 전환: 유저 모드에서 커널 모드로 진입하고 다시 나오는 과정.  
  
본 코드는 멀티코어 환경에서 실행될 경우 각 프로세스가 서로 다른 코어에 할당되어 병렬로 실행될 수 있다.  
해결 방법 : sched_setaffinity() 시스템콜을 사용하여 모든 프로세스가 동일한 CPU코어에서만 실행되도록 하였다.

## 결론  
실험을 통해 다음을 확인할 수 있었다.  
  
1. **Context Switch Overhead**: 프로세스 전환은 즉각적이지 않으며, 시간 비용이 발생한다.  
2. **System Call에 비해** 프로세스 상태(register, stack 등)를 보존하고 복구해야하는 **Context Switch의 비용이 훨씬 크다**.  
3. 따라서 잦은 문맥 전환은 시스템 전체의 처리량을 저하시킬 수 있음을 이해하였다.
